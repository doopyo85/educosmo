1️⃣ 방향 설정은 정확하다
“완성 게임 ❌ / 짧은 장면 ⭕”

당신이 말한 이 전환이 결정적이에요.

완성 게임을 만들게 하지 말고
완성 게임에 들어가는 ‘장면 하나’를 만든다

이건 단순 축소가 아니라 인지 부담을 완전히 바꾸는 설계입니다.

아이 기준에서의 차이

❌ 완성 게임
→ “언젠가 끝날 것 같은 거대한 것”

⭕ 장면 하나
→ “오늘 끝낼 수 있는 것”

👉 기억 유지 가능성 + 성취감이 급격히 올라갑니다.

2️⃣ “리메이크”보다 “장면 단위 구현”이 더 좋은 이유
🔧 기존 프로젝트 리메이크의 한계

구조를 이해하지 못하면 그냥 복붙

바꿔도 왜 되는지 모름

실패하면 전체가 무너짐

🎬 장면 단위 구현의 장점

시작과 끝이 명확

실패해도 그 장면만 실패

“이 장면은 내가 만들었다”는 소유감

👉 아이에게 작가 의식이 생깁니다.

3️⃣ 수업 시간 조건을 정확히 반영한 설계 원칙

당신의 조건:

주 1회 90분 또는

주 2회 60분

지난주 기억이 약함

👉 이 조건에서 지켜야 할 절대 원칙 3가지

✅ 원칙 1

한 수업 = 하나의 장면 = 하나의 결과

90분이면 → 장면 완성

60분 × 2회면 →

1회차: 만들기

2회차: 살 붙이기

❌ “다음 주에 이어서” 금지
⭕ “다음 주엔 다른 장면”

✅ 원칙 2

지난주 코드를 ‘기억’하게 하지 말 것

이게 정말 중요합니다.

아이의 기억을 믿지 말고
코드가 스스로 설명하게 만들어야 합니다.

그래서 장면 코드는 반드시:

파일 하나

함수 1~2개

변수 이름이 상황 설명

def buy_item():
    if gold >= price:
        gold -= price
        inventory.append(item)


👉 코드가 복습 자료

✅ 원칙 3

매 수업은 독립 실행 가능해야 한다

지난주 코드 없어도 실행 가능

새 파일로 시작 가능

“이거 지난주에 했잖아” ❌

👉 수업은 에피소드형이어야 합니다.

📘 이벤트 중심 → 객체지향 전환 코딩수업 교수설계 명세서
1. 설계 목적 (Purpose)

본 교수설계는 다음의 문제를 해결하는 것을 목표로 한다.

아이들이 완성된 프로젝트는 원하지만,
그 결과를 만들기 위한 개념 학습에는 동기를 잃는 문제

주 1회 90분 / 주 2회 60분 수업 구조에서
기억 단절, 이어가기 실패가 반복되는 문제

객체지향(OOP)을 설명해도 받아들이지 못하는 문제

👉 목표는 OOP를 가르치는 것 자체가 아니라,
아이 스스로 *“객체가 필요하다고 느끼는 순간”*을 만들어
자연스럽게 객체지향 사고를 탑재하는 것이다.

2. 기본 철학 (Educational Philosophy)
2.1 학습자 인지 구조 전제

아이들의 사고 흐름은 다음과 같다:

의미(원하는 결과)
→ 사건(무슨 일이 일어나야 하는가)
→ 맥락(언제, 어떤 상황에서)
→ 존재(누가 있었어야 하는가)


즉, 존재(Object)는 선결 조건이 아니라 사후 구성물이다.

객체를 먼저 요구하면, 인지는 즉시 붕괴한다.

2.2 객체지향(OOP)의 위치 규정

OOP는 사고의 출발점이 아니다

OOP는 **코드가 복잡해졌을 때 등장하는 ‘정리 도구’**이다

❌ “이건 객체야”
⭕ “이렇게 하니까 너무 헷갈리지? 묶어보자”

3. 교수설계 핵심 전략 (Core Strategy)
3.1 결과 중심 → 구조 후행 설계 (Backward Design)

모든 수업은 다음 순서를 따른다.

눈에 보이는 결과 먼저

그 결과를 만드는 이벤트

이벤트에 필요한 조건과 상태

상태가 많아졌을 때 정리 필요성 발생

이때 객체(OOP) 도입

3.2 장면(Scene) 중심 콘텐츠 구성

완성 게임 ❌

완성 게임을 구성하는 ‘장면 하나’ ⭕

예:

상하좌우 이동만 구현

상점에서 물건 구매만 구현

화면 전환 이벤트만 구현

👉 한 수업 = 하나의 장면 = 하나의 성취

4. 시간 구조 설계 (Time Structure)
4.1 2주 단위 마일스톤 설계

마일스톤 정의

2주 안에 아이가
“나 지금 뭔가 만들고 있다”고 느낄 수 있는
최소 기능 단위

4.2 왜 2주인가

1주 → 성취감 부족

1달 → 동기 소멸

2주 → 아이가 버틸 수 있는 최대 미래 보상 거리

5. 수업 운영 원칙 (Operation Rules)
원칙 1. 한 수업은 반드시 독립 실행 가능해야 한다

지난주 코드 없이 실행 가능

새 파일로 시작 가능

“지난주 기억”을 전제하지 않는다

원칙 2. 기억은 아이가 아니라 시스템이 담당한다

파일명 = 장면 설명

원칙 3. 미완·지저분한 코드는 허용된다

복붙 ⭕

버그 ⭕
